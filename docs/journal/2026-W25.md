### 요청 1 — 중대 발견으로 착수 중단 (분석 불일치 → 보고)
- ADR 0014의 승인된 접근(same-site 서브도메인 `api-chronoshealth.ever-day.com`)이 **직접 provision 불가**.
- 이유: `ever-day.com` 네임서버는 **Gabia(ns1.gabia.co.kr)** — 존이 Cloudflare에 없음(ADR 0006). CF Worker 커스텀 도메인/라우트는 존이 CF에 있어야 함.
- **수정 단일안(DNS 변경 없음)**: 게이트웨이를 `chronoshealth.ever-day.com/api/*` 경로로 **same-origin**, CF **Pages Function 프록시**로 노출 → 쿠키가 first-party가 되어 ITP 회피. 기존 Pages 프로젝트에 추가, Gabia/존 이전 불필요.
- ADR 0014 슬라이스(서브도메인 → same-origin 프록시) 개정 필요. 게이트웨이 코드는 작성/배포하지 않음(파괴적 변경 회피). 승인 대기.

### 🛑 접속 불가 사고 — Pages Git 자동빌드 원인 규명·비활성화 (2026-06-20)
- **증상**: 커스텀 도메인 `chronoshealth.ever-day.com`이 `/`조차 404(빈 응답) — 접속 불가.

- **임시 복구**: 정상 로컬 `out/` 수동 재배포(6ac2a7df)로 덮어 즉시 정상화(전 경로 200 실측).
- **근본 해결**: CF API로 `source.config.deployments_enabled=false`/`production_deployments_enabled=false`/`preview_deployment_setting="none"` PATCH → 자동빌드 비활성화. 이후 push는 배포 미트리거, 배포는 수동 `wrangler pages deploy out`만. 메모리 [[feedback_chronos_pages_git_autobuild_broken]] 기록.

- **원인(실측)**: `middleware/rate-limit.ts` 의 카운터가 **pseudonym 단독 키 + 24시간 창**. 즉 모든 rateLimit 라우트가 **하루치 카운터 1개를 공유**. 메시지 화면이 GET 메시지 4초 폴링 + markRead + (신규) unread 20초 폴링을 호출 → 공유 카운터가 send 한도(120/day)를 빠르게 잠식 → 전송 429. 재배포 시 isolate 메모리 초기화로 잠깐 정상→재발(="또 에러") 패턴 일치. 라이브 재현(전송 API 자체는 200)으로 백엔드 정상 확인 후 rate-limit 으로 좁힘.
- **수정**: 미들웨어 인스턴스별 고유 버킷 키(라우트별 독립 카운터) + `windowMs` 파라미터. 메시징 폴링/전송/조회 라우트(send 60/min, messages GET 120/min, conversations/detail/unread/read)를 분당 창으로. 설문·AI 등 quota 라우트는 기존 1일 창 유지.
- 검증: 게이트웨이 **259 테스트 통과**. **라이브 재현 검증** — 동일 사용자가 GET 메시지 130회(과거 send 한도 120 초과 유도) 후 전송 → HTTP 200(과거 로직이면 429). 커밋 `acc325c`.
- 배포: 게이트웨이 Worker(`28e648d7`). 웹 변경 없음. releases `v0.0.10`. 테스트 계정 정리 완료.
