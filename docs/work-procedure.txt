[Heading1] 0. 요약 (Executive Summary)
크로노스 헬스는 사용자의 정적·동적·환경 데이터를 통합해 '생물학적 나이(Biological Age)'와 '질병 위험 지수'를 산출하고, 데이터 주권을 사용자에게 돌려주는 '건강 기여 보상(Health-to-Earn)' 모델을 결합한 디지털 트윈 헬스케어 플랫폼이다. 본 문서는, 본 프로젝트의 첫날부터 LBank 상장까지의 작업 절차를 단계별로 정리한 운영 매뉴얼이다.
운영의 네 가지 핵심 원칙은 다음과 같다.
[ListParagraph] 명세 우선 개발(Spec-Driven) : 모든 기능은 명세서에서 시작한다. 클로드 코드에 코드를 시키기 전에 '명세 → 아키텍처 → 작업 분해' 순으로 합의한 뒤 구현에 진입한다.
[ListParagraph] 수직 슬라이싱(Vertical Slicing) : '예측 한 줄을 끝까지' 먼저 완성한다. 인프라·모델·UI·저장·리포트로 이어지는 얇은 한 줄을 1주 안에 동작시키고, 그 위에 살을 붙인다.
[ListParagraph] 설계 단계 규제 반영(Compliance-by-Design) : 미국 HIPAA, 유럽 GDPR, 한국의 의료법·개인정보보호법·특정금융정보법 요건을 코드 작성 전 단계의 데이터 모델에 반영한다. 후속 리팩토링 비용이 가장 큰 영역이다.
[ListParagraph] 재현 가능성(Reproducibility) : 모든 머신러닝 실험과 스마트 컨트랙트 배포는 깃 태그 + 실행 해시로 재현 가능해야 한다. 외부 감사 직전에 '커밋이 어디였더라' 같은 사고를 막는 유일한 방법이다.

[Heading2] 2.2 데이터 계층 (의료 데이터 격리)
[ListParagraph] 개인식별정보(PII)는 별도 '신원 금고(identity-vault)' 서비스에서만 다루고, 분석 DB에는 의사식별자(user_pseudonym_id)만 존재한다.
[ListParagraph] 원본 의료 데이터는 KMS로 암호화된 S3 영역에 저장하고, 분석 단계로 갈수록 익명화 수준을 강화한다 (브론즈 → 실버 → 골드).
[ListParagraph] 모든 데이터 접근은 감사 로그(audit_log) 테이블에 기록되며, 추가만 가능하고 7년간 보관한다.
[ListParagraph] '잊혀질 권리' 요청은 신원 금고의 매핑만 끊으면 분석 DB 데이터는 통계적 익명 상태로 잔존이 가능하다. 단, 법무 검토를 거쳐야 한다.
[Heading2] 2.3 머신러닝 계층
[ListParagraph] 피처 스토어 : Feast 기반, 온라인(Redis)/오프라인(Parquet on S3) 분리.
[ListParagraph] 모델군 : (a) PhenoAge 계열 회귀 — 생물학적 나이, (b) 트랜스포머 시계열 — 12개월 위험 변화, (c) 생존분석(DeepHit) — 사건별 위험률(hazard).
[ListParagraph] 실험 관리 : MLflow + DVC. 모든 실험은 데이터셋 해시 + 설정 + 깃 SHA 세 가지로 식별한다.
[ListParagraph] 서빙 : Triton 또는 BentoML, A/B 트래픽 분할은 API 게이트웨이에서 결정.
[ListParagraph] 드리프트 모니터링 : Evidently AI로 PSI/KS 지표 측정, 임계치 초과 시 PagerDuty와 슬랙으로 알림.
[Heading2] 2.4 웹3 계층
[ListParagraph] 체인 선택 : 1차 Polygon PoS(가스비 저렴, EVM 호환), 추후 Base·Arbitrum 확장 검토.
[ListParagraph] 핵심 컨트랙트 : ChronosToken(ERC-20), DataVault(사용자 데이터 메타 해시 보관), RewardDistributor(데이터 기여 보상 정산), ZKVerifier(건강 증명 검증).
[ListParagraph] 영지식 회로 : Circom으로 'BMI ∈ [a,b]', 'HRV ≥ X' 같은 단순 술어부터 시작. 회로 복잡도와 증명 생성 시간을 모바일에서 측정해 한계를 설정.
[ListParagraph] 자금 관리 : Gnosis Safe(3-of-5 멀티시그), 컨트랙트 업그레이드는 OpenZeppelin Defender + 48시간 타임락(TimeLock)으로 강제.

[Heading2] 3.2 루트 CLAUDE.md 운영
CLAUDE.md는 클로드 코드가 매 세션 자동으로 읽는 '프로젝트의 상수'를 담는다. 클라우드 브릿지의 교훈은 이 파일을 '짧고 단호하게' 유지하는 것이다. 길어질수록 클로드가 덜 따른다.
[ListParagraph] 프로젝트 한 줄 요약 + 현재 단계 (예: '단계 2 - 웨어러블 수집')
[ListParagraph] 절대 규칙 (예: '개인정보는 분석 DB 스키마에 절대 넣지 않는다', '모든 외부 호출은 lib/http로만 한다')
[ListParagraph] '먼저 읽어라' 링크 : docs/adr/, docs/spec/current.md
[ListParagraph] 작업 시작 시 체크리스트 (테스트 먼저, 타입 우선, 변경 전 플랜 모드)
[ListParagraph] 금지 사항 (main에 강제 푸시, 비밀키 커밋, 마이그레이션 없는 스키마 변경)
도메인별 CLAUDE.md는 해당 폴더 안에 두고, 그 폴더에서 작업할 때만 자동으로 로드되게 한다 (예: contracts/CLAUDE.md에 Foundry·Slither 규칙).

[Heading2] 3.4 서브에이전트 운영 원칙
일반 작업은 메인 컨텍스트에서 진행하되, 다음 네 가지는 항상 서브에이전트로 분리한다.
[ListParagraph] 컴플라이언스 리뷰어 : PR 단위로 자동 실행. 개인정보 누출 패턴, 로깅 위반, 감사 로그 누락 점검.
[ListParagraph] 솔리디티 감사관 : contracts/ 변경 시 Slither·Aderyn 실행 결과와 변경 의도의 정합성 확인.
[ListParagraph] 머신러닝 실험가 : 새 실험 요청 시 데이터셋 검증 → 학습 → 평가 → MLflow 기록까지 단발 위임.
[ListParagraph] 테스트 작성자 : 메인 세션이 구현을 끝낸 직후, 테스트 커버리지 보강만 별도로 위임.
클라우드 브릿지에서 학습한 핵심 — 서브에이전트는 결과를 '요약'으로만 받고 코드를 그대로 신뢰하지 않는다. 메인 에이전트가 변경 이력을 다시 읽어보고 합친다.

[ListParagraph] pre-commit : gitleaks, eslint --max-warnings 0, ruff, mypy --strict, forge fmt --check, Slither(컨트랙트 변경 시), prettier.

[Heading1] 4. 개발 절차 (명세 → 슬라이스 → 출시)
[Heading2] 4.1 표준 사이클
[ListParagraph] 포착 : Linear 이슈 또는 docs/spec/draft/에 한 문장 문제 정의.
[ListParagraph] 명세 : /spec 명령 → docs/spec/<기능>.md (목표 / 비목표 / API / 데이터 / 위험 / 미해결 질문).
[ListParagraph] 계획 : 클로드 코드의 플랜 모드(Plan Mode)로 작업 분해. 사람이 승인할 때까지 코드 변경 금지.
[ListParagraph] 슬라이스 : /slice 명령 → 가장 얇은 수직 슬라이스 하나를 1~3일 내 완료할 단위로 쪼갠다.
[ListParagraph] TDD : 실패하는 테스트 → 구현 → 통과 → 리팩토링. 메인 에이전트가 진행하고 테스트 작성자 에이전트가 보강.
[ListParagraph] 셀프 리뷰 : 클로드에게 '본인의 변경 이력을 외부 시각으로 비판하라' 요청 + 컴플라이언스 리뷰어 동시 호출.
[ListParagraph] PR : 자동 템플릿 (명세 링크 / 마이그레이션 영향 / 보안 영향 / 롤백 절차).
[ListParagraph] 리뷰 : 사람 1명 + 클로드의 /review 슬래시 커맨드.
[ListParagraph] 머지 : 스쿼시 머지, 컨벤셔널 커밋. main은 항상 그린(green) 상태.
[ListParagraph] 출시 : 카나리 5% → 25% → 100%. 지표는 그라파나 대시보드 자동 링크.
[ListParagraph] 결정 기록 : 비가역적 결정은 docs/adr/NNNN-*.md로 기록. /adr 명령 사용.
[Heading2] 4.2 작업 유형별 처리 방식
작업 유형
클로드 코드 운영 방식
UI 변경
플랜 없이 직접 진행 가능. Storybook + 시각 회귀(Playwright) 필수.
API 추가
OpenAPI 명세 우선 → 코드 생성. 클라이언트 SDK는 자동 생성.
DB 마이그레이션
플랜 모드 강제. 마이그레이션 → 백필 → 코드 배포 → 마이그레이션 정리 네 단계 분리 PR.
ML 변경
머신러닝 실험가 서브에이전트. 모델 PR에는 학습 데이터 해시·평가 리포트 첨부 필수.
스마트 컨트랙트
플랜 모드 + 솔리디티 감사관 + 변경 라인 단위 가스 비교 + 테스트넷 배포 후 사람 최종 승인.
보안·개인정보 영역
컴플라이언스 리뷰어 + 사람 2명 리뷰. 다른 작업과 같은 PR에 섞지 않는다.

[Heading2] 4.3 깃 워크트리(Worktree) 멀티 트랙
클라우드 브릿지에서 효과를 본 패턴 — 동시에 진행되는 작업이 세 개 이상이 되면 깃 워크트리로 디렉토리를 분리하고, 각 워크트리마다 별도의 클로드 코드 세션을 띄운다. 컨텍스트가 섞이지 않고, 한 트랙이 실패해도 다른 트랙은 계속 흐른다.
git worktree add ../chronos-ml-v2     feature/ml-pheno-age-v2 git worktree add ../chronos-contracts feature/reward-distributor git worktree add ../chronos-hotfix    hotfix/oura-rate-limit
[Heading2] 4.4 컨텍스트 위생
[ListParagraph] 세션이 길어지면 /compact 또는 명시적 요약 후 새 세션을 시작한다.
[ListParagraph] 작업이 끝나면 docs/journal/YYYY-MM-DD.md에 결정·실패·미해결 이슈를 자동 기록.
[ListParagraph] 매주 금요일 문서 큐레이터 에이전트가 결정 기록·명세 업데이트 PR을 생성하면 사람이 머지.

[ListParagraph] 모델 PR은 학습 데이터 해시, 학습 설정, 평가 리포트(데이터 슬라이스별)를 첨부해야 머지 가능.

[Heading2] 6.3 보안 기준선
[ListParagraph] 비밀키 : SOPS + age, GitHub Actions에는 키 없는(OIDC) 방식으로만 주입. .env 파일 커밋 금지.

[Heading1] 12. 맺음말
크로노스 헬스는 기술의 깊이와 윤리의 무게가 동시에 요구되는 프로젝트다. 폼코치에서 사용자 경험을, 투윈즈닷컴에서 데이터 모델링을, 클라우드 브릿지에서 운영 안정성을 익혔다면, 이 프로젝트의 새 영역은 '의료·금융 규제'와 '신뢰의 운영'이다. 빠르게 만들고 싶을수록 결정 기록에 근거를 적고, 작은 PR로 머지하고, 컴플라이언스 영역은 빨리 가지 말 것.
클로드 코드는 이 작업의 가장 빠른 동료가 될 수 있지만, 의사결정과 책임은 사람의 몫이다. 본 문서는 그 협업의 표준을 정의한다.
